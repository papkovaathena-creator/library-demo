# Задания на доработку — Часть 2: Gradle + OpenAPI + Библиотеки

---

## Зачем всё это: общая картина

В первой части заданий мы привели в порядок код **внутри** приложения: транзакции, валидация, обработка ошибок, тесты. Это навыки написания кода.

Во второй части мы выходим за рамки кода и учимся работать с **инфраструктурой разработки** — тем, что окружает код и определяет, как он собирается, описывается и переиспользуется. Это три отдельных навыка, каждый из которых встретится на любом реальном проекте:

**Gradle** — умение управлять сборкой проекта. Maven справляется с простыми случаями, но как только появляются кодогенерация, несколько модулей, условная логика сборки — XML становится обузой. Gradle с Kotlin DSL даёт полноценный язык программирования вместо конфигурационного файла. Не «Maven или Gradle» — на практике нужны оба, и выбор зависит от проекта.

**OpenAPI и кодогенерация** — умение работать по контракту. В командной разработке бэкенд и фронтенд (или два микросервиса) не могут ждать друг друга. OpenAPI-спецификация — это договор, который фиксирует API до написания кода. Из этого договора автоматически генерируются интерфейсы, модели и документация. Если реализация отклоняется от контракта — код не компилируется. Это называется API-first подход, и это стандарт в индустрии.

**Библиотеки и модули** — умение разделять проект на переиспользуемые части. В реальном мире API-контракт нужен не только серверу: его подключают клиенты, тесты, другие сервисы. Выделение контракта в отдельную библиотеку — первый шаг к модульной архитектуре и микросервисам.

К концу этих заданий у тебя будет multi-module Gradle проект, где API описан спецификацией, код генерируется автоматически, контракт живёт в отдельной библиотеке, а компилятор гарантирует, что реализация соответствует контракту.

## Часть A: Миграция с Maven на Gradle

### Почему Gradle, а не оставить Maven

Maven и Gradle решают одну задачу — собрать проект. Но делают это по-разному:

| | Maven | Gradle |
|---|---|---|
| **Конфигурация** | XML (`pom.xml`) — декларативный, жёсткая структура | Kotlin/Groovy DSL (`build.gradle.kts`) — программируемый |
| **Скорость сборки** | Каждый раз выполняет все фазы | Инкрементальная сборка + build cache — пересобирает только изменённое |
| **Управление зависимостями** | Один скоуп `compile` на всё | Гранулярные конфигурации: `implementation`, `api`, `runtimeOnly`, `compileOnly` |
| **Многомодульность** | Работает, но конфигурация через наследование (`<parent>`) и copy-paste | Первоклассная поддержка: `include()`, `project()`, `subprojects {}` |
| **Кодогенерация** | Плагины с XML-конфигурацией, часто громоздко | Задачи (tasks) — обычные функции, легко настраивать и связывать |
| **Кривая обучения** | Проще начать, но сложно кастомизировать | Сложнее начать, но гибче в долгосрочной перспективе |

Maven отлично работает для простых проектов. Но как только нужна кодогенерация (OpenAPI), несколько модулей, условная логика — Gradle проще и выразительнее. Весь Android, сам Spring Boot, Netflix OSS, LinkedIn — используют Gradle. Знать оба инструмента обязательно.

---

### 1. Инициализировать Gradle-обёртку

**Что сделать:** Выполнить `gradle init` (или вручную создать файлы `build.gradle.kts`, `settings.gradle.kts`, а также каталог `gradle/wrapper/` с `gradle-wrapper.jar` и `gradle-wrapper.properties`). После этого проект должен запускаться через `./gradlew build`.

**Зачем:** Gradle — второй по популярности инструмент сборки в Java-экосистеме. В отличие от Maven с его XML, Gradle использует Kotlin DSL — полноценный язык программирования. Это позволяет писать условную логику, переиспользовать код и проще читать конфигурацию. Многие компании и проекты (Android, Spring Boot сам по себе) используют Gradle. Уметь работать с обоими инструментами — обязательный навык.

---

### 2. Перенести зависимости из `pom.xml` в `build.gradle.kts`

**Что сделать:**
- Подключить плагины: `java`, `org.springframework.boot`, `io.spring.dependency-management`.
- Перенести все зависимости, соблюдая скоупы:
  - `implementation` — для основных зависимостей (starter-webmvc, starter-data-jpa, starter-security, starter-validation, starter-actuator, flyway, postgresql)
  - `runtimeOnly` — для H2 и PostgreSQL-драйвера
  - `testImplementation` — для тестовых зависимостей (spring-boot-starter-test, assertj)
- Указать `java.toolchain.languageVersion = JavaLanguageVersion.of(17)`.
- Убедиться, что `./gradlew build` проходит, все тесты зелёные.

**Зачем:** Главное отличие от Maven — вместо `<scope>test</scope>` в Gradle есть гранулярные конфигурации: `implementation`, `api`, `runtimeOnly`, `testImplementation`, `testRuntimeOnly`, `compileOnly`. Понимание, какая зависимость куда идёт, — ключ к чистому classpath и быстрой сборке. Например, H2 нужен только в runtime — нет смысла тащить его в compile-time classpath.

**Подсказка:** В Maven `spring-boot-starter-parent` задавал версии через BOM. В Gradle ту же роль играет плагин `io.spring.dependency-management` — он подтягивает BOM Spring Boot и позволяет не указывать версии стартеров вручную.

---

### 3. Настроить Flyway в Gradle

**Что сделать:** Убедиться, что Flyway-миграции работают с обоими профилями (hsqldb для тестов, postgres для прода). В `build.gradle.kts` никакой специальной настройки Flyway обычно не нужно — достаточно зависимости `flyway-core`. Проверить, что `./gradlew test` проходит с H2, а `./gradlew bootRun` поднимается с PostgreSQL.

**Зачем:** Flyway — это то место, где при миграции на Gradle чаще всего ломается сборка. Ученик должен убедиться, что автоконфигурация Spring Boot подхватывает миграции без ручного вмешательства.

---

### 4. Удалить `pom.xml`

**Что сделать:** После того как `./gradlew build` проходит зелёным — удалить `pom.xml`. Убедиться, что IDE переиндексировала проект как Gradle-проект (IntelliJ: File -> Invalidate Caches / Reimport). Закоммитить изменения.

**Зачем:** Два инструмента сборки в одном проекте — источник путаницы. Кто-то запустит `mvn test`, кто-то `./gradlew test`, и результат может отличаться. Одна система сборки — один источник правды.

## Часть B: OpenAPI — контракт и кодогенерация

### Почему API-first, а не code-first

Есть два подхода к разработке API:

**Code-first** (то, что сейчас в проекте): сначала пишем контроллер, потом (может быть) описываем API в документации. Проблемы:
- Документация устаревает на второй день, потому что никто не обязан её обновлять
- Фронтенд-разработчик ждёт, пока бэкенд напишет код, чтобы понять формат ответов
- Два сервиса договариваются "на словах", а потом на интеграции выясняется, что один шлёт `bookName`, а другой ждёт `name`
- Нет способа автоматически проверить, что реализация соответствует договору

**API-first** (куда переходим): сначала пишем спецификацию (OpenAPI YAML), из неё генерируем интерфейсы и модели, потом реализуем. Преимущества:
- **Параллельная работа**: фронт и бэк стартуют одновременно — у обоих есть контракт
- **Компилятор как страж**: если реализация не соответствует контракту — код не скомпилируется. Не тесты упадут, не QA найдёт на staging, а компилятор не даст собрать
- **Единый источник правды**: спецификация → интерфейсы, модели, документация (Swagger UI), клиентские SDK. Всё генерируется из одного файла
- **Версионирование контракта**: спецификация лежит в git, изменения видны в diff, можно ревьюить API отдельно от реализации
- **Кроссплатформенность**: из одной спецификации можно сгенерировать клиент на TypeScript, Kotlin, Python, Go — для любого потребителя

В индустрии API-first — стандарт для любого проекта, где API потребляют больше одного клиента. Это не «модный подход», а инженерная практика, которая экономит часы на интеграции и баг-фиксах.

---

### 5. Написать OpenAPI-спецификацию для существующего API

**Что сделать:** Создать файл `src/main/resources/openapi/library-api.yaml` в формате OpenAPI 3.0. Описать в нём все эндпоинты:

| Метод | Путь | Описание |
|-------|------|----------|
| GET | /books | Список книг с фильтрацией и пагинацией |
| GET | /books/{id} | Получить книгу по ID |
| POST | /books | Создать книгу |
| PUT | /books/{id} | Обновить книгу |
| DELETE | /books/{id} | Удалить книгу |
| POST | /books/{id}/reserve | Забронировать книгу |
| POST | /books/{id}/return | Вернуть книгу |

Для каждого эндпоинта описать:
- Параметры запроса (path, query)
- Тело запроса (request body) со схемой
- Все варианты ответов (200, 201, 204, 400, 404, 409)
- Схемы моделей: `BookDto`, `ErrorResponse` (в разделе `components/schemas`)

**Зачем:** OpenAPI — это **контракт** между бэкендом и клиентом. Писать его руками — важный навык, потому что:
1. Заставляет продумать API **до** написания кода (API-first подход)
2. Спецификация — единый источник правды: из неё генерируется код, документация и тесты
3. Фронтенд-разработчик может начать работу сразу, не дожидаясь готового бэкенда — у него есть контракт

**Подсказка:** Для пагинированного ответа `GET /books` используй параметры `page`, `size`, `sort` как query-параметры. Ответ опиши как объект с полями `content` (массив `BookDto`), `totalElements`, `totalPages`, `number`, `size` — это соответствует структуре `Page<T>` из Spring Data.

---

### 6. Подключить `openapi-generator` плагин в Gradle

**Что сделать:** В `build.gradle.kts`:
1. Добавить плагин `org.openapi.generator` (актуальная версия — 7.x)
2. Настроить задачу `openApiGenerate`:

```kotlin
openApiGenerate {
    generatorName.set("spring")
    inputSpec.set("$rootDir/src/main/resources/openapi/library-api.yaml")
    outputDir.set("${layout.buildDirectory.get()}/generated")
    apiPackage.set("ru.athena.library_demo.api.generated")
    modelPackage.set("ru.athena.library_demo.api.generated.model")
    configOptions.set(mapOf(
        "interfaceOnly" to "true",
        "useSpringBoot3" to "true",
        "useTags" to "true",
        "dateLibrary" to "java8",
        "openApiNullable" to "false",
        "skipDefaultInterface" to "false"
    ))
}
```

3. Добавить сгенерированные исходники в source set:
```kotlin
sourceSets {
    main {
        java.srcDir("${layout.buildDirectory.get()}/generated/src/main/java")
    }
}
```

4. Сделать так, чтобы `compileJava` зависел от `openApiGenerate`:
```kotlin
tasks.compileJava {
    dependsOn(tasks.openApiGenerate)
}
```

5. Добавить зависимости, которые нужны сгенерированному коду:
   - `implementation("io.swagger.core.v3:swagger-annotations:2.2.x")`
   - `implementation("jakarta.validation:jakarta.validation-api")`

6. Убедиться, что `./gradlew build` проходит — в `build/generated` появляются интерфейс API и модели.

**Зачем:** Идея code-first («сначала пишем код, потом документируем») приводит к тому, что документация устаревает на второй день. API-first с кодогенерацией решает это: спецификация **является** кодом. Если разработчик изменит контроллер, не обновив спецификацию, — проект не скомпилируется, потому что контроллер больше не реализует сгенерированный интерфейс.

---

### 7. Реализовать сгенерированный интерфейс в контроллере

**Что сделать:**
1. `LibraryController` должен реализовать (`implements`) сгенерированный интерфейс (например, `BooksApi`).
2. Удалить ручные аннотации `@GetMapping`, `@PostMapping` и т.д. с методов контроллера — они уже описаны в сгенерированном интерфейсе.
3. Оставить `@RestController` и `@RequestMapping("/books")` на классе (или перенести базовый путь в спецификацию).
4. Привести сигнатуры методов в соответствие с интерфейсом.

**Зачем:** Контроллер теперь реализует контракт. Если кто-то добавит новый эндпоинт в спецификацию — компилятор заставит добавить реализацию. Если кто-то изменит сигнатуру в контроллере — компиляция сломается. Контракт энфорсится на уровне типов, а не "по договорённости".

---

### 8. Заменить ручной `BookDto` на сгенерированную модель

**Что сделать:**
1. Удалить класс `ru.athena.library_demo.api.dto.BookDto`.
2. Во всех местах, где использовался `BookDto`, заменить на сгенерированную модель из `ru.athena.library_demo.api.generated.model`.
3. Обновить `BookMapper` — маппинг между `Book` (entity) и сгенерированной моделью.
4. Обновить тесты — они должны использовать новые DTO.
5. Убедиться, что `./gradlew build` проходит, все тесты зелёные.

**Зачем:** Дублирование моделей — путь к рассинхрону. Если спецификация говорит, что поле называется `title`, а ручной DTO говорит `name` — это баг, который найдут только на интеграции. Одна модель, один источник.

**Ловушка:** Валидационные аннотации (`@NotBlank`, `@Size`) теперь задаются в спецификации через `minLength`, `maxLength`, `pattern` — генератор сам поставит Jakarta Validation аннотации на сгенерированные поля. Проверь, что валидация по-прежнему работает.

---

### 9. Убедиться, что всё работает вместе

**Что сделать:**
1. `./gradlew clean build` — сборка проходит без ошибок
2. Все существующие тесты зелёные
3. Приложение поднимается через `./gradlew bootRun`
4. Ручная проверка через curl/Postman:
   - `GET /books` возвращает пагинированный список
   - `POST /books` с невалидным телом возвращает 400
   - `DELETE /books/{id}` забронированной книги возвращает 409
5. Сгенерированные файлы **не** закоммичены в git (добавить `build/` в `.gitignore`)

**Зачем:** Миграция сборки и кодогенерация — это рефакторинг инфраструктуры. Поведение системы не должно измениться. Тесты — гарантия, что ничего не сломалось. Правило: сгенерированный код — это артефакт сборки, как `.class` файлы. Его не коммитят, его воспроизводят.

## Часть C: Выделение API-контракта в отдельную библиотеку

### Почему отдельная библиотека, а не всё в одном модуле

После части B у нас OpenAPI-генерация работает, но всё лежит в одном модуле. Кажется, что этого достаточно. Но представь реальную ситуацию:

**Сценарий 1: Второй сервис.** Появляется `notification-service`, который должен знать, как выглядит `BookDto`, чтобы отправлять уведомления о бронировании. Если `BookDto` генерируется внутри `library-demo` — notification-service придётся подключить **весь** library-demo как зависимость: Spring Security, JPA, Flyway, PostgreSQL-драйвер. Ему ничего из этого не нужно — ему нужен только DTO.

**Сценарий 2: Фронтенд.** Фронтенд-разработчик хочет сгенерировать TypeScript-клиент из той же спецификации. Если YAML лежит внутри серверного приложения — ему придётся клонировать весь репозиторий и разбираться в структуре, чтобы найти один файл.

**Сценарий 3: Контрактное тестирование.** QA-команда пишет интеграционные тесты. Им нужны модели для десериализации ответов. Опять же — подключать весь сервер ради DTO?

**Решение:** выделить контракт (спецификацию + сгенерированные интерфейсы/модели) в отдельный модуль `library-api`. Этот модуль:
- Содержит **только** контракт — никакой бизнес-логики, никакой БД
- Собирается в лёгкий JAR (десятки КБ вместо десятков МБ)
- Подключается как зависимость кем угодно: сервером, другим сервисом, тестами, клиентом
- Версионируется отдельно — можно выпустить новую версию контракта до того, как сервер его реализует

Это паттерн **shared API module**, который используется практически в каждом микросервисном проекте. В Spring-экосистеме его часто называют `*-api` или `*-client` модулем. Например: `order-service-api`, `payment-client`.

Заодно мы освоим **multi-module Gradle** — навык, без которого невозможно работать в проекте крупнее одного микросервиса.

---

### 10. Превратить проект в multi-module Gradle

**Что сделать:**
1. В корневом `settings.gradle.kts` объявить два модуля:
```kotlin
rootProject.name = "library-demo"
include("library-api")
include("library-app")
```
2. Создать структуру каталогов:
```
library-demo/              ← корневой проект (только settings.gradle.kts и корневой build.gradle.kts)
├── library-api/           ← модуль-библиотека (контракт)
│   ├── build.gradle.kts
│   └── src/main/resources/openapi/library-api.yaml
├── library-app/           ← модуль-приложение (реализация)
│   ├── build.gradle.kts
│   └── src/main/java/...  ← весь текущий код
```
3. Перенести исходный код приложения в `library-app/src/`.
4. Корневой `build.gradle.kts` должен содержать только общие настройки (версию Java, репозитории), которые применяются к обоим подпроектам через `subprojects {}` или `allprojects {}`.
5. Убедиться, что `./gradlew build` из корня собирает оба модуля.

**Зачем:** В реальных проектах API-контракт и реализация живут отдельно. Причина простая: контракт нужен **не только** серверу. Его подключают фронтенд (для TypeScript-клиента), мобильное приложение, другие микросервисы, интеграционные тесты в соседних командах. Если контракт лежит внутри серверного приложения — всем этим потребителям придётся тащить к себе Spring, JPA, Flyway и всё остальное. Отдельная библиотека — это чистый контракт без лишнего багажа.

Multi-module проект — стандарт в любом проекте крупнее pet project. Понимание `settings.gradle.kts`, `subprojects`, зависимостей между модулями — обязательный навык для работы в команде.

---

### 11. Настроить модуль `library-api`

**Что сделать:** В `library-api/build.gradle.kts`:
1. Подключить плагины: `java-library` (не `java`!) и `org.openapi.generator`.
2. Перенести сюда задачу `openApiGenerate` и OpenAPI-спецификацию из задания 6.
3. Перенести зависимости, нужные сгенерированному коду (`swagger-annotations`, `jakarta.validation-api`, `spring-web`).
4. Настроить source sets и зависимость `compileJava` от `openApiGenerate` — аналогично заданию 6.
5. **Не подключать** плагин `org.springframework.boot` — этот модуль не приложение, а библиотека. Он не должен уметь запускаться.
6. Убедиться, что `./gradlew :library-api:build` проходит и генерирует JAR с интерфейсами и моделями.

**Зачем:** Ключевой момент — плагин `java-library` вместо `java`. Разница:
- `java` предоставляет конфигурацию `implementation` — зависимости видны **только** внутри модуля
- `java-library` добавляет конфигурацию `api` — зависимости, объявленные как `api`, попадают в compile-time classpath **потребителей** библиотеки

Пример: библиотека зависит от `swagger-annotations`. Если объявить её как `implementation` — код приложения не увидит эти аннотации. Если как `api` — увидит. Для аннотаций на публичных интерфейсах нужен `api`.

Это одно из самых практичных отличий Gradle от Maven, где такой гранулярности просто нет.

**Подсказка:** Зависимости модуля `library-api` от Spring (`spring-web`) для аннотаций `@RequestMapping`, `@PathVariable` и т.д. нужно объявить как `api`, а не `implementation`, чтобы приложение видело эти типы при реализации интерфейса.

---

### 12. Подключить `library-api` к `library-app`

**Что сделать:** В `library-app/build.gradle.kts`:
1. Добавить зависимость на модуль-библиотеку:
```kotlin
dependencies {
    implementation(project(":library-api"))
    // ... остальные зависимости приложения
}
```
2. Удалить из `library-app` всё, что переехало в `library-api`:
   - OpenAPI-спецификацию
   - Плагин `openapi-generator`
   - Зависимости `swagger-annotations`, `jakarta.validation-api` (они придут транзитивно через `library-api`)
   - Ручной `BookDto` (если ещё не удалён в задании 8)
3. `LibraryController` по-прежнему `implements BooksApi` — но теперь интерфейс приходит из JAR-файла модуля `library-api`.
4. Убедиться, что `./gradlew :library-app:build` проходит, тесты зелёные.

**Зачем:** Теперь контроллер реализует интерфейс, который **физически** находится в другом модуле. Это значит:
- Контракт можно версионировать отдельно от реализации
- Если завтра понадобится второй сервис, который вызывает библиотеку — он подключит тот же `library-api` и получит готовые модели, не зная ничего о реализации
- В будущем `library-api` можно опубликовать в корпоративный Nexus/Artifactory — и любая команда сможет его подключить как обычную зависимость

Это паттерн, который используется в каждом микросервисном проекте. Сервис A предоставляет `service-a-api` с DTO и Feign/RestClient-интерфейсами, сервис B подключает его и вызывает как обычный Java-метод.

---

### 13. Проверить полную сборку

**Что сделать:**
1. `./gradlew clean build` из корня — оба модуля собираются, тесты зелёные.
2. `./gradlew :library-app:bootRun` — приложение стартует.
3. Ручная проверка: все эндпоинты работают как раньше.
4. Проверить порядок сборки: `./gradlew build --dry-run` — `library-api` должен собираться **до** `library-app` (Gradle разрешает это автоматически через граф зависимостей).
5. `.gitignore` обновлён: `**/build/` исключён для обоих модулей.

**Зачем:** Multi-module проект добавляет сложность в сборку. Gradle определяет порядок задач через DAG (directed acyclic graph): если `library-app` зависит от `library-api`, Gradle гарантирует, что сначала соберёт библиотеку. Но нужно убедиться, что это действительно работает — `--dry-run` показывает план без выполнения.

## Бонус (необязательно)

### 14. Добавить Swagger UI

**Что сделать:** В модуле `library-app` подключить зависимость `springdoc-openapi-starter-webmvc-ui`. Настроить путь к спецификации. Открыть `http://localhost:8080/swagger-ui.html` и убедиться, что все эндпоинты отображаются с описаниями, параметрами и примерами.

**Зачем:** Swagger UI — это интерактивная документация, которую можно "потрогать руками". Вместо чтения YAML-файла или расспрашивания бэкенд-разработчика, любой участник команды открывает страницу в браузере и видит:
- Все эндпоинты с описаниями
- Формат запросов и ответов с примерами
- Кнопку "Try it out" — можно отправить реальный запрос прямо из документации

Это радикально сокращает время онбординга новых разработчиков и снижает количество вопросов вида "а какой формат у этого эндпоинта?". В API-first подходе Swagger UI генерируется из той же спецификации, из которой генерируется код — поэтому документация **всегда** актуальна. Это невозможно при ручном ведении документации.

### 15. Опубликовать `library-api` в Maven Local

**Что сделать:**
1. В `library-api/build.gradle.kts` подключить плагин `maven-publish`.
2. Настроить публикацию:
```kotlin
publishing {
    publications {
        create<MavenPublication>("mavenJava") {
            from(components["java"])
            groupId = "ru.athena"
            artifactId = "library-api"
            version = "0.0.1-SNAPSHOT"
        }
    }
}
```
3. Выполнить `./gradlew :library-api:publishToMavenLocal`.
4. Проверить, что JAR появился в `~/.m2/repository/ru/athena/library-api/`.
5. Попробовать в `library-app` заменить `project(":library-api")` на координаты из Maven Local:
```kotlin
implementation("ru.athena:library-api:0.0.1-SNAPSHOT")
```
и убедиться, что сборка проходит.
6. **Вернуть обратно** `project(":library-api")` — для разработки удобнее project-зависимость.

**Зачем:** Это задание показывает полный жизненный цикл библиотеки — от кода до артефакта, который могут подключить другие.

В реальном проекте библиотеки не остаются внутри multi-module проекта навсегда. Рано или поздно они публикуются в **репозиторий артефактов** — Nexus, Artifactory, GitHub Packages или Maven Central. После публикации любая команда в компании подключает библиотеку одной строкой в `build.gradle.kts`, не клонируя исходники и не зная деталей реализации. `publishToMavenLocal` — это локальная репетиция этого процесса.

Понимание координат (`groupId:artifactId:version`), разницы между `project(":module")` и внешней зависимостью, и самого процесса публикации — обязательный навык для работы в командах, где больше одного репозитория.
